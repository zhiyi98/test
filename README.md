```
数据库优化的七个步骤：
1.sql的优化
2.索引的优化：七字口诀
3.表结构优化，一定要选用使用的字段属性
4.事务的处理
5.锁表的问题
6.系统配置的优化，操作系统的连接数、断开连接的资源回收的设置、打开文件的数量的限制、MqSQL的连接数的设置、内存设置、慢查询设置等等
7.硬件的优化，服务器优化、网络带宽等等，主要是运维的工作。

```



```
sql的优化:
    1、少使用select * ，尽量使用具体字段；
    2、对于条件来说等号之类两边的字段类型要一致，字符串不加单引号索引会失效；
    3、尽量少使用Order By 排序，对于需要多个字段进行排序的可以使用组合索引；
    4、对于group by 语句要先过滤后分组；
    5、在查询时减少使用null，对字段有多个null的可以加默认值；
    6、少使用like，对于需要使用的， 如需要使用尽量用 like abc%这种，不要把%放字段前面；
    7、在where后面少使用函数或者算数运算；
    8、去除的distinct 过滤字段要少，避免 distinct * ；
    9、不要超过5个以上的表连接。
    10、不要使用not in,这个会破坏索引，not in 可以用not exists 来代替;
    11、使用连接（join）来代替子查询；
```



```
索引失效的问题：
七字口诀：
模：使用like关键字的时候，以%开头，索引会失效。
型：如果数据类型错了，比如字段类型是varchar，where条件用number，索引也会失效。
数：对索引的字段使用内部函数，索引也会失效。这种情况应该简历基于函数的索引。
空：索引不存储空值，如果索引不限制列为not null，数据库会认为有可能存在空值，所以不会按照索引进行运算。
运：对索引列进行加减乘除等运算，会导致索引失效。
最：在复合索引中索引列的顺序至关重要，如果不是按照缩影的最左列开始查找，则无法使用索引。
快：如果数据库预计使用全表扫描要比使用索引快，则不使用索引。
```



```
联合索引：
联合索引：指对一张表的多个字段同时简历一个索引，提高查询速度。要是使用联合索引，需要遵循最左前缀原理。
最左前缀原理：就是联合索引的事件是有顺序的，而不能中断。比如说：有name,age,sex三个字段，那我们在查询的时候就应该使用name,age,sex三个字段全指匹配。如果中间的age字段去掉了，那么只能用到最左边的name，而后面的sex因为age去掉了，所以只能用到最左边的name，这就是最左前缀原理。
```

### 四.乐观锁/悲观锁/锁的优化/分布式锁

```
  为了解决并发的问题，mysql引入了锁的概念。
  从阻塞的角度将可以分为乐观锁和悲观锁，乐观锁可以理解为cas+循环，根据版本号不断的判断是否是记录中相应的旧值，如  果是就修改，如果不是就取出最新的值作为旧值继续判断比较并替换。这样可以避免阻塞的发生，但是这样也会造成cpu有些时候无畏的消耗。
  第二个是悲观锁，主要就是会造成阻塞，mysql中有读锁和写锁两种都属于悲观锁。读锁就是这一行加了读锁，其他线程也可以加读锁，但是不能加写锁。也就是说对读锁是开放的，对写锁是封闭的，也叫共享锁，写锁也叫排他锁。如果一个线程在这条记录上加了写锁，那么其他线程要加读锁或者是写锁都是不允许的，就会造成阻塞。
  如果按照锁定的范围来分的话，可以分为行锁、表锁和页锁。
  锁是基于索引的，如果时行锁的话，其实他就是直接锁定索引上面的一个叶里面的那个节点，就是锁定的行记录。接下来就是表锁，也就时锁定的整张表。最后是页锁也可以叫间隙锁，比如说我的主键是整数，那么此时有一条记录是1，有一条记录是3，那么里面不存在的2就是间隙锁，此时我锁定了1-3的范围，那么也无法向这个范围内插入2这条记录的。接下来就是临建锁，可以理解为间隙锁+行锁，也就是说刚才的例子，那么我的2和3加起来就可以是一个临建锁。
  根据锁的范围基本可以划分为这些，然后在锁的过程中，如果存在着循环嵌套锁，可能会造成死锁的情况。比如说一个线程持有了A的锁却在等待B的锁，另外一个线程持有了B的锁却在等待A的锁，那么此时两个线程就会一直阻塞，因为他们都互相持有着对方的锁，但是又不释放而等待对方的锁，就会一直阻塞下去。在mysql中也提供了一种解决死锁的机制检测死锁，如果检测到死锁，那么就会终端一个线程，让另外一个线程执行。当然如果mysql中存在大量死锁的情况，那么就会出现不断的终端线程，最后造成cpu无畏的消耗，就相当于一次都没做成，一直在那里释放线程。
```

### 七.==/equals()/hashcode()

```
==: 它的作用是判断两个对象的地址是不是相等。即，判断两个对象是不是同一个对象。

(基本数据类型 == 比较的是值，引用数据类型 == 比较的是内存地址)

equals(): 它的作用也是判断两个对象是否相等。

情况1：类没有覆盖 equals() 方法。

情况2：类覆盖了 equals() 方法。比如：String中的equals方法是被重写过的。因为object的equals方法是比较的对象的内存地址，而String的equals方法比较的是对象的值。

hashCode() 的作用是获取哈希码，也称为散列码，hashCode() 定义Object中，这就意味着Java中的任何类都包含有hashCode()函数。它的特点是：能根据key速的检索出对应的value，即可以根据key快速找到所需要的对象。
```



```
hashCode()与equals()的的区别

如果两个对象相等，则hashcode一定也是相同的

两个对象相等，对两个对象分别调用equals方法都返回true

两个对象有相同的hashcode值，它们也不一定是相等的

因此，equals 方法被覆盖过，则 hashCode 方法也必须被覆盖.
```





### 八.String/StringBuffer/StringBuilder

```
##### 可变性

String类中使用字符数组保存字符串，private　final　char　value[]，所以string对象是不可变的。StringBuilder与StringBuffer都继承自AbstractStringBuilder类，在AbstractStringBuilder中也是使用字符数组保存字符串，char[] value，这两种对象都是可变的。

##### 线程安全性

StringBuilder 底层是 数组

StringBuilder 底层是 数组+锁

String中的对象是不可变的，也就可以理解为常量，线程安全。AbstractStringBuilder是StringBuilder与StringBuffer的公共父类，定义了一些字符串的基本操作，如expandCapacity、append、insert、indexOf等公共方法。StringBuffer对方法加了同步锁或者对调用的方法加了同步锁，所以是线程安全的。StringBuilder并没有对方法进行加同步锁，所以是非线程安全的。

##### 性能

每次对String 类型进行改变的时候，都会生成一个新的String对象，然后将指针指向新的String 对象。StringBuffer每次都会对StringBuffer对象本身进行操作，而不是生成新的对象并改变对象引用。相同情况下使用StirngBuilder 相比使用StringBuffer 仅能获得10%~15% 左右的性能提升，但却要冒多线程不安全的风险。

##### 对于三者使用的总结

如果要操作少量的数据用 = String

单线程操作字符串缓冲区 下操作大量数据 = StringBuilder

多线程操作字符串缓冲区 下操作大量数据 = StringBuffer

#### String类的常用方法有哪些？

indexOf()：返回指定字符的索引。
charAt()：返回指定索引处的字符。
replace()：字符串替换。
trim()：去除字符串两端空白。
split()：分割字符串，返回一个分割后的字符串数组。
getBytes()：返回字符串的 byte 类型数组。
length()：返回字符串长度。
toLowerCase()：将字符串转成小写字母。
toUpperCase()：将字符串转成大写字符。
substring()：截取字符串。
equals()：字符串比较。
```
